# RPD 写作规范（用于指导设计与开发）

> 目的：统一需求文档质量，让产品、设计、开发、测试基于同一份文档协作，减少返工。

## 1. 文档定位

- `RPD`（Requirement + Product Design）是需求与设计一体化文档。
- 一份 RPD 必须回答 4 个问题：做什么、为什么做、怎么做、如何验收。
- 适用范围：新功能、功能迭代、体验改版、流程优化。

## 2. 必填结构（按顺序）

每份 RPD 必须包含以下章节，不可缺失：

1. **背景与目标**
   - 业务背景、用户痛点、当前问题
   - 目标指标（如转化率、完成率、留存、时长）
2. **范围定义**
   - In Scope（本次做）
   - Out of Scope（本次不做）
3. **用户与场景**
   - 目标用户画像
   - 典型使用场景（至少 2 个）
4. **功能需求（FR）**
   - 按用户流程拆分功能点
   - 每个功能点包含：触发条件、输入、处理逻辑、输出、异常处理
5. **非功能需求（NFR）**
   - 性能、稳定性、可用性、兼容性、安全性、可维护性
6. **交互与视觉设计要求**
   - 信息架构、页面/模块结构
   - 关键交互状态（默认/加载/空态/错误/成功）
   - 设计约束（色彩、字号、间距、组件规范）
7. **数据与埋点**
   - 关键事件、事件字段、上报时机
   - 指标口径说明
8. **验收标准（AC）**
   - 使用 Given / When / Then 或编号条款描述
   - 每条标准必须可测试、可判定
9. **发布与回滚**
   - 上线策略（灰度/全量）
   - 风险项与回滚条件

## 3. 写作质量标准

- **明确性**：避免“尽量、可能、优化一下”等模糊词。
- **可执行性**：开发和设计无需口头补充即可开工。
- **可验证性**：每条需求必须映射到验收标准。
- **可追踪性**：目标 -> 功能 -> 埋点 -> 验收可一一对应。

## 4. 需求条目模板（必须使用）

```md
### FR-x 功能名称
- 目标：<该功能解决的核心问题>
- 角色：<谁在用>
- 前置条件：<触发前状态>
- 主流程：
  1) ...
  2) ...
- 异常流程：
  - 网络失败：...
  - 数据为空：...
- 输出结果：<页面/数据变化>
- 验收标准：
  - AC-x1: Given ... When ... Then ...
  - AC-x2: ...
```

## 5. 设计指导要求（重点）

为保证 RPD 可以直接指导设计，文档必须补齐：

- **页面级目标**：每个页面写清主任务与次任务。
- **布局草图描述**：至少说明模块顺序与优先级。
- **状态完整性**：默认、加载、空态、错误、成功 5 类状态必须定义。
- **组件复用策略**：优先使用现有组件；新增组件需说明复用边界。
- **可访问性要求**：对比度、键盘可达、语义标签、错误提示可读。
- **响应式规则**：桌面/平板/移动端的布局变化规则。

## 6. 变更管理

- RPD 变更必须记录：变更时间、变更人、变更原因、影响范围。
- 涉及交互或接口变更时，必须同步更新验收标准与测试点。

## 7. 评审清单（提交前自检）

- [ ] 目标是否量化且可衡量？
- [ ] 范围是否清晰区分做/不做？
- [ ] 功能与异常流程是否完整？
- [ ] 是否可直接产出 UI 设计稿？
- [ ] 埋点是否覆盖关键路径？
- [ ] 验收标准是否可执行、可测试？
- [ ] 发布与回滚是否可落地？

---

**执行规则**：从即日起，所有需求文档按本规范撰写；缺失“设计指导要求”或“验收标准”的文档不得进入开发排期。
